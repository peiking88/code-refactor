# Code Refactor

基于证据的系统化代码审查与重构方法论 —— Claude Code 技能。

> **版本:** 1.2.0

## 概述

帮助开发者在重构前进行**五轴快速扫描**（正确性、可读性、架构、安全、性能），基于调用者计数和类型安全边界做出**有数据支撑的决策**，避免凭直觉重构。同时提供**异常审计方法论**（排查该用未用、不该用却用、异常类使用不当三类问题）、**技术债评估框架**和**系统化代码审查流程**，覆盖代码/架构/异常/测试/文档/基础设施六大维度，以 ROI 为导向制定优先修复路线图。

## 技能触发

当用户提出以下请求时自动触发：

- "review this code" / "审查这段代码"
- "refactor this" / "重构这个"
- "simplify this code" / "简化这段代码"
- "clean this up" / "清理一下这个模块"
- "is this too complex" / "这个是不是太复杂了"
- "audit this file" / "审计这个文件"
- "audit exception handling" / "审计异常处理"
- "check exception usage" / "检查异常使用"
- "find missing try/catch" / "查找缺失的异常处理"
- "create a debt remediation plan" / "制定技术债修复计划"
- "prioritize tech debt cleanup" / "优先处理技术债"

## 内容覆盖

### 核心原则

1. **行为保持** —— 重构改变结构而非外部行为
2. **小步提交** —— 每次提交一个逻辑变更，之间运行测试
3. **证据优先** —— 实施前统计调用者、展示差异、陈述权衡
4. **简单胜于聪明** —— 内联琐碎代码，只为值得的抽象提取
5. **不重构测试** —— 测试是行为的规格说明

### 十大代码坏味道

| 编号 | 坏味道 | 重构策略 |
|------|--------|----------|
| 1 | 长方法 (>50行) | 提取方法 / 提取到委托类 |
| 2 | 重复代码 (>5行相同) | 模板方法 / 策略模式 |
| 3 | 参数过多 (>4个) | 参数对象 / 构建器模式 |
| 4 | 基本类型偏执 | 替换为值对象 / 派生类型 |
| 5 | 过度复杂 | 内联可有可无的抽象 |
| 6 | 霰弹式修改 | 内聚相关行为 |
| 7 | 特性依恋 | 将方法移至正确的类 |
| 8 | 不恰当亲密 | 提取层 / 打破循环 |
| 9 | 被拒绝的遗赠 | 组合优于继承 |
| 10 | 注释过多 | 重命名使代码自解释 |

### 语言参考

方法论正文（SKILL.md）语言无关；`references/` 目录下提供 Python、TypeScript、Java 的 Before/After 具体语法示例，覆盖类型安全边界、异常审计、参数对象、分支合并、设计模式等。

### 技术债评估

| 维度 | 扫描内容 |
|------|----------|
| 代码债 | 重复逻辑、复杂方法、深层嵌套、上帝类 |
| 架构债 | 缺失/泄漏抽象、违反边界、循环依赖 |
| 异常债 | 缺失错误边界、吞没异常、过度宽泛 catch、异常类型不当、异常驱动控制流 |
| 测试债 | 未覆盖路径、缺失集成测试、脆弱测试 |
| 文档债 | 未记录的公共 API、缺失架构图 |
| 基础设施债 | 手动部署、无回滚、缺失监控 |

对每项债务量化影响（工时/成本/频率），以 ROI 排序制定快速见效 / 中期 / 长期路线图，并建立预防策略（质量门禁、债务预算）。

## 安装使用

1. 将本仓库克隆到 Claude Code 技能目录
2. 运行 `/check-skills` 验证技能加载正常
3. 在任何代码库中请求代码审查或重构即可自动触发

## 何时不应该重构

- 代码不会再变更
- 没有测试覆盖（先写测试）
- 在紧迫的截止日期下
- 与 Bug 修复混在一起（先修 Bug 再重构）
- "只是觉得该重构"而无明确目的
- 不理解代码为何存在（切斯特顿围栏原则）

## 输出格式

每次重构结果包含三部分：

```
1. WHAT changed（改了什么，一句话）
2. BEFORE/AFTER diff（前后差异对比）
3. TRADEOFF（变好了什么，变差了什么）
```
